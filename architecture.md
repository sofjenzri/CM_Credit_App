# Refactored Architecture

## Overview

This repository now runs as a two-part web application:

- `maestro-ui/`: the active React + Vite frontend used by credit analysts
- `backend/`: the active Node.js + Express backend that exposes the application API and integrates with UiPath

The root `package.json` only provides convenience scripts to launch the two subprojects.

The application is case-management oriented:

- the frontend authenticates users with UiPath OAuth
- the backend retrieves case data, task data, stage data, documents, and metadata from UiPath
- the frontend displays case lists, case details, analyst dashboards, and a loan request form

## Active Runtime

### Frontend

The frontend entrypoint is:

- `maestro-ui/src/main.tsx`
- `maestro-ui/src/App.tsx`

It uses:

- React
- React Router
- Tailwind-based styling
- local `fetch`-based service modules for backend communication

The main active routes are:

- `/login`
- `/oauth-callback`
- `/dashboard`
- `/submissions`
- `/cases/:id`
- `/new-request`

Several dashboard-style pages are still demo-oriented, but they are part of the active UI shell.

### Backend

The backend runtime is now split into:

- `backend/server.js`: bootstrap only
- `backend/app.js`: Express app composition
- `backend/routes/*`: API route definitions
- `backend/lib/*`: UiPath access, mapping, entity operations, case aggregation

This replaced the previous monolithic `server.js` behavior with a modular structure.

## Backend Module Responsibilities

### `backend/server.js`

Responsible only for:

- creating the app
- starting the HTTP server
- logging startup mode

### `backend/app.js`

Responsible for:

- creating the Express app
- registering middleware
- mounting all `/api` routers
- serving `maestro-ui/dist` when a production frontend build exists
- returning a simple root response when the frontend build is absent

### `backend/config/uipath.js`

Centralizes UiPath-related environment configuration:

- base URL
- org name
- tenant name
- folder key
- client credentials
- target process key
- target case model id
- Data Fabric entity names

It also exposes helpers to determine:

- whether UiPath mode is active
- whether client credentials are configured

### `backend/routes/system.js`

Lightweight system endpoints:

- `GET /api/health`
- `GET /api/source`

### `backend/routes/cases.js`

Case-management endpoints:

- `POST /api/debug/uipath-data`
- `GET /api/cases`
- `GET /api/cases/:id`

These endpoints switch between:

- mock mode using `backend/mock/data.js`
- real UiPath-backed mode using `fetchUiPathData()`

### `backend/routes/documents.js`

Document streaming endpoint:

- `GET /api/documents/:recordId`

This endpoint retrieves attachments from UiPath Data Fabric and streams them back to the frontend.

### `backend/routes/loan-requests.js`

Loan request creation endpoint:

- `POST /api/loan-requests`

This route:

- parses multipart form uploads
- validates the incoming loan request
- resolves the target Data Fabric entities
- creates the main case record
- creates document records
- uploads document attachments

### `backend/routes/oauth.js`

OAuth exchange endpoint:

- `POST /api/oauth/token`

This is used by the frontend callback page to exchange the UiPath authorization code for an access token.

### `backend/lib/uipath-client.js`

Low-level UiPath HTTP helpers:

- bearer token extraction
- client credential token retrieval and caching
- URL construction
- JSON request wrappers
- generic request wrapper

### `backend/lib/fetch-uipath-data.js`

Main aggregation layer for case data.

It:

- reads process instances from PIMS
- resolves target entities from Data Fabric
- loads main case records and document records
- enriches each case with stages, tasks, chronology, and document links
- returns:
  - case list data for `/api/cases`
  - detail data for `/api/cases/:id`

### `backend/lib/case-processors.js`

Transforms raw UiPath case-management data into frontend-friendly structures:

- stages
- tasks
- activity timeline
- case detail payloads

### `backend/lib/entity-operations.js`

Encapsulates Data Fabric write/read operations:

- read with fallback
- create entity record
- update entity record
- insert then update case id
- upload attachment
- map business fields onto detected entity columns

### `backend/lib/data-mappers.js`

Contains reusable normalization and extraction helpers used across the backend.

Examples:

- token normalization
- field discovery
- SLA normalization
- entity record helpers
- generic array extraction from UiPath responses

## Frontend Module Responsibilities

### `maestro-ui/src/services/oauth.ts`

Implements the active authentication flow:

- starts OAuth PKCE login with UiPath Identity
- handles callback exchange through backend `/api/oauth/token`
- stores access tokens in local storage
- exposes auth helper functions for route protection and display

### `maestro-ui/src/services/cases.ts`

Implements backend API access for:

- case list retrieval
- case detail retrieval

It also attaches the stored bearer token when present.

### `maestro-ui/src/hooks/useCaseDetail.ts`

Loads:

- one case detail
- the full case list for previous/next navigation

### `maestro-ui/src/hooks/useDocumentViewer.ts`

Handles:

- authenticated document fetching
- temporary blob URL creation
- document preview state
- download-from-viewer behavior

### `maestro-ui/src/utils/caseFormatters.ts`

Shared formatting and display helpers:

- dates
- amounts
- status translation
- badge styling
- UiPath Maestro deep links
- document URL normalization

## Main Application Flows

### 1. Authentication Flow

1. User opens `/login`
2. Frontend starts UiPath OAuth PKCE flow
3. UiPath redirects to `/oauth-callback`
4. Frontend sends code to backend `/api/oauth/token`
5. Backend exchanges the code with UiPath Identity
6. Frontend stores the access token and unlocks protected routes

### 2. Case List Flow

1. User opens `/submissions`
2. Frontend calls `GET /api/cases`
3. Backend fetches and aggregates case data from UiPath or mock data
4. Frontend renders the case table, filters, and portfolio widgets

### 3. Case Detail Flow

1. User opens `/cases/:id`
2. Frontend calls `GET /api/cases/:id`
3. Backend returns:
   - client data
   - credit data
   - stages
   - tasks
   - chronology/activity
   - document references
4. Frontend renders the detailed case workspace

### 4. Document Flow

1. User opens a document from the case detail page
2. Frontend calls the backend document URL
3. Backend fetches the attachment from UiPath Data Fabric
4. Frontend displays the downloaded blob in the viewer

### 5. Loan Request Creation Flow

1. User fills the form on `/new-request`
2. Frontend submits multipart form data to `POST /api/loan-requests`
3. Backend validates input
4. Backend creates the main case in Data Fabric
5. Backend creates document records and uploads files
6. Frontend displays the created case id and upload results

## Runtime Modes

### Mock Mode

Activated when:

- UiPath base configuration is incomplete, or
- `USE_MOCK_DATA=true`

Used for:

- local demo behavior without live UiPath dependency
- UI testing against predictable data

### UiPath Mode

Activated when:

- base UiPath configuration is present

Authentication can use:

- an incoming bearer token from the frontend
- backend client credentials if configured

## Static Frontend Serving

The backend no longer points to the old `frontend/` directory.

Instead:

- in development, frontend and backend are typically run separately
- in production-style mode, the backend serves `maestro-ui/dist` if it exists

This removes the mismatch that previously existed between the live backend and the inactive old frontend folder.

## Legacy and Archived Code

The following are not part of the active runtime:

- `__archive__/`
- previously unused frontend runtime variants that were removed during refactoring

The archive folder is kept as historical material only.

## Suggested Development Workflow

### Frontend development

Run:

```bash
cd maestro-ui
npm run dev
```

### Backend development

Run:

```bash
cd backend
npm run dev
```

### Production-style frontend build

Run:

```bash
cd maestro-ui
npm run build
```

After this, the backend can serve the built frontend from `maestro-ui/dist`.

## Current Technical Notes

- The frontend build is valid and produces `maestro-ui/dist`
- The backend modular routing is active and tested
- Some UI pages are intentionally demo/static and not yet driven by backend APIs
- The production build currently emits a large chunk warning for the main frontend bundle, which is not blocking but may be optimized later
